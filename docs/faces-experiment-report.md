# Faces & Metadata Round-Trip Experiment — Full Report

**Project:** `immich-cli` (standalone HTTPX + Click uploader with XMP sidecar)
**Date:** 2026-07-19 → 2026-07-20
**Author:** B. I. Czaczkes (user) + Pi (documentation)
**Status:** In progress — story so far, not finished.

---

## 0. Purpose

We wanted to know, definitively, what happens to an image's **faces, people
names, ratings ("stars"), GPS, description, and tags** when:

1. we attach an **XMP sidecar** generated by `immich-cli`, and
2. we upload it to an **Immich server**, and
3. we later **download it again** — does Immich keep what we sent?

The motivating mystery was that **uploaded faces kept coming back as
`faces: 0`** on a live server, even though the download path clearly *writes*
MWG-RS face regions into the sidecar. This report traces how we found out why,
and what the sidecar actually does and does not achieve.

All evidence lives in this repo:
- `trace.log` — the CLI trace of every upload/download run.
- `New/` — the **source** image + sidecar that was uploaded (`New/19580128.jpg`,
  `New/19580128.jpg.xmp`).
- `downloads/` — what Immich handed back on each download.

---

## 1. The starting observation (2026-07-19)

We downloaded two real family photos from a live Immich server using the new
`immich-cli download` command:

- `19580128.jpg` (asset `00b19997-…`, 2560×3412)
- `19580328.jpg` (asset `02cee462-…`, 2560×3480)

Each download produced a `.meta.json` and an `.xmp` sidecar. Two things jumped
out:

**(a) Ratings ("stars") were NOT where we expected.** They were absent from the
top level of the `.meta.json`, but present under `exif.rating` (value `5` for
the first image, `null` for the second). The XMP sidecar carried them as
`<xmp:Rating>5</xmp:Rating>`. So rating *is* captured on download — just nested
in EXIF, not a first-class key. (On upload, Immich's async metadata-extraction
overwrites `rating` from the file's own EXIF, so a `PUT /assets/{id}` rating
returns `null` — a separate, known upload-side issue.)

**(b) The face size mismatch.** The `.meta.json` face regions were referenced
against a **1440×1919** image, while the asset itself was **2560×3412**:

| Field | 19580128.jpg |
|---|---|
| Asset dimensions (`width` / `exif.image_width`) | **2560 × 3412** |
| Face region `image_width` / `image_height` | **1440 × 1919** |
| XMP `mwg-rs:AppliedToDimensions` | **2560 × 3412** |

The asset was flagged `resized: true`. Immich stores a downscaled preview and
ran its ML face detection on **that preview (1440×1919)**, then recorded the
boxes against the preview size. The download's XMP, by contrast, re-normalizes
the boxes by the **original** dimensions and records `AppliedToDimensions =
2560×3412`.

We verified the math: a face box center on the 1440-wide preview
`(345+754)/2 / 1440 = 0.3816` exactly equals the XMP normalized `stArea:x =
0.381597`. So the download export is internally consistent; the boxes are just
referenced to different sizes in the two representations.

**Hypothesis at this point:** maybe the upload-side `faces: 0` was because our
upload XMP used `AppliedToDimensions = original (2560×3412)` while Immich
expects preview-sized (1440×1919) coordinates. We set out to test uploading
with the XMP sidecar and watching what comes back.

---

## 2. Cleaning the server (so the experiment is uncontaminated)

To get a clean baseline we needed the two test people gone:

- **Alfred Czaczkes** → person ID `020b92cd-ac8b-487a-b726-e79706fc81d1`
- **Benjamin Czaczkes** → person ID `13df863a-20fb-490a-8851-734a0cfc80ae`

We wrote `scripts/check_people_cleared.py` (read-only: `GET /people` +
`GET /faces` for those IDs/names). It confirmed the server was clear
("nothing found → CLEARED").

> Lesson: deleting just the *image* is not enough — Immich keeps the *person*
> and recreates faces from the file. You must delete all images of a person,
> then `DELETE /people/{id}`.

Separately, we wrote `scripts/pause_jobs.py` / `scripts/resume_jobs.py` to
control Immich's background job queues via `PUT /jobs/{name}` with
`{"command": ..., "force": true}` — mirroring the desktop app's
`NetworkClient.set_job_command`. The desktop pauses six queues:
`facialRecognition, faceDetection, smartSearch, thumbnailGeneration,
metadataExtraction, videoConversion`.

> Gotcha we hit: the OpenAPI spec *mentions* `SidecarCheck`, `SidecarWrite`,
> `SidecarQueueAll`, but those are **not valid `PUT /jobs/{name}` queue names**
> — the server rejects them with `400 invalid_value`. There is no pausable
> "sidecar" queue. Sidecar ingestion rides on `metadataExtraction` (see §4).

---

## 3. Run 1 — upload with sidecar, jobs stopped, then released (MISLEADING)

On Windows, with all jobs paused, we uploaded `19580128.jpg` (auto-discovered
`.xmp` sidecar). We saw faces present, then released all jobs and the image
ended up with **4 faces** (2 named + 2 unnamed). We initially concluded "the
XMP sidecar is ingested even when jobs are stopped."

**This conclusion was wrong** — disproved by Run 2.

---

## 4. Run 2 — upload + download with ALL jobs paused (the correction)

Same upload (`19580128.jpg` + sidecar, asset `6dd908df-…`), then with **all
nine queues paused** we downloaded. The asset came back **completely empty**:

- `GET /assets/{id}` → `exifInfo` all `null` (width/height, date, GPS,
  description, rating), `tags: []`, `people: []`, `width/height: null`.
- `GET /faces?id=` → `[]` (0 faces).
- `GET /albums?assetId=` → `[]` (0 albums).

**Correction:** the XMP sidecar is **NOT** ingested synchronously at upload
time. With all jobs paused, *nothing* is extracted — not even the sidecar.
Sidecar (and EXIF) ingestion is a **background job** (`metadataExtraction`,
which also processes sidecars). Earlier "faces seen with jobs stopped" must
have had a job still running.

So there is no way to make Immich apply the sidecar without running at least
the metadata/sidecar jobs. Pausing *everything* yields an empty asset.

---

## 5. Run 3 — DEFINITIVE run on a blank server (metadata ON, ML OFF)

To remove all doubt, the user **physically deleted the Immich server from disk
and recreated it** — a truly blank instance with no prior memory. Then:

1. `pause_jobs.py` was run (all six standard queues paused).
2. Uploaded `New/19580128.jpg` **with its XMP sidecar** (asset
   `6dd908df-b133-42b8-a0a0-31e81d2ff40d`, `201 created`).
3. Released **only `metadataExtraction`** (so the sidecar/EXIF is read) but
   kept `faceDetection` / `facialRecognition` / `smartSearch` **paused** (so no
   ML face detection runs).
4. Downloaded the asset again.

### 5.1 What came back (`downloads/19580128.jpg.meta.json`, download at 03:09)

| Field | Value | Ingested from sidecar? |
|---|---|---|
| `width` / `height` | 2560 × 3412 | yes (EXIF) |
| `exif.dateTimeOriginal` | 1959-01-28 | yes |
| `gps` | 31.793263, 35.216578 | yes |
| `location` | Wadi al-Joz / West Bank / Palestine | yes |
| `description` | "9 month old me and my dad" | yes |
| `exif.rating` | **5** | yes (`<xmp:Rating>`) |
| `tags` | Benjamin, Alfred (server-created IDs) | yes (`dc:subject`) |
| `people` | Alfred Czaczkes, Benjamin Czaczkes | yes (`PersonInImage`) |
| `albums` | `[]` | not run this time (no `--album`) |
| `faces` | **4** | see below |

**The sidecar is fully ingested.** Tags, GPS, date, rating, description,
people names — all present, all correct. The XMP sidecar is the real source of
truth, exactly as suspected.

### 5.2 The faces — and the remaining puzzle

`GET /faces?id=` returned **4 faces**, every one `sourceType: "exif"` (NOT
`machine-learning`), each linked to the correct named person:

| # | Person | Box (on 2560×3412) | sourceType |
|---|---|---|---|
| 1 | Alfred Czaczkes | (613,627)→(1340,1671) | exif |
| 2 | Alfred Czaczkes | (613,627)→(1340,1671) | exif |
| 3 | Benjamin Czaczkes | (1514,887)→(2033,1557) | exif |
| 4 | Benjamin Czaczkes | (1514,887)→(2033,1557) | exif |

Two findings:

**(i) Coordinates are correct and original-sized.** Alfred's center
`(613+1340)/2 / 2560 = 0.3814` matches the source sidecar's normalized
`stArea:x = 0.381597` (and the re-exported XMP `0.381445`). Because our
`AppliedToDimensions` was the *original* `2560×3412`, Immich stored the boxes
against the original, not a preview. **The earlier "preview vs original"
dimension worry from §1 did NOT block ingestion** — original-sized
`AppliedToDimensions` works. (The 1440×1919 boxes in §1 came from ML running on
a resized copy; with ML off, the sidecar's original-sized regions are honored.)

**(ii) Each sidecar face is DOUBLED.** The uploaded sidecar had **exactly 2
regions** (Alfred + Benjamin, one each — verifiable in `New/19580128.jpg.xmp`
and the trace's uploaded-sidecar dump). The server returned **4 faces**, with
Alfred's box duplicated verbatim and Benjamin's box duplicated verbatim. So
Immich ingested the 2 sidecar regions but **wrote 4 faces** — a server-side
duplication at the `exif` source level, not ML.

This is the *new* duplicate-faces mechanism, distinct from Run 1's "2 named +
2 unnamed ML" duplication. Run 1 had ML re-detection on top of the sidecar; Run
3 isolates the pure sidecar path and shows the sidecar regions themselves get
doubled by Immich's ingestion.

### 5.3 The re-exported sidecar confirms it

`downloads/19580128.jpg.xmp` (regenerated by `meta_export` from the server
state) contains **4 `<rdf:li>` regions** — Alfred ×2, Benjamin ×2 — matching
the 4 server faces exactly. So the doubling is faithfully round-tripped:
2 in → 4 out → 4 back.

## 6. Conclusions so far

1. **The XMP sidecar IS the source of truth.** With `metadataExtraction` (which
   also ingests sidecars) enabled, Immich picks up tags, GPS, date, rating,
   description, people names, **and** face regions — no ML required. The JSON
   `.meta.json` is only a derived debug snapshot; the sidecar is what travels.

2. **Sidecar ingestion is a background job, not synchronous.** Pause
   `metadataExtraction` (+ sidecar processing) and the asset is empty. This is
   why naive uploads (job paused or not yet run) show `faces: 0`.

3. **Faces from the sidecar are stored `sourceType: "exif"`**, linked to the
   named people — not as ML detections. Good.

4. **But Immich doubles each sidecar face** → 2 real faces become 4 stored
   faces (verbatim duplicate boxes, same person, `exif` source). This is the
   current blocker for a clean round-trip. Root cause is server-side ingestion,
   not our XMP (our 2 regions go in; 4 come out).

5. **Original-sized `AppliedToDimensions` is correct.** No need to switch to
   preview-sized coordinates — the boxes round-trip accurately against the
   original image dimensions.

6. **Albums were not applied in Run 3** (the CLI run was upload+download only,
   no `--album`). The sidecar does carry `immich:albums`, but we didn't verify
   album ingestion this run.

---

## 7. Open questions / next steps

- **Why does Immich double sidecar faces?** Is it a known quirk of
  `metadataExtraction` re-processing the sidecar, or specific to our MWG-RS
  structure (e.g. both `mwg-rs:Name` and `iptc-core:PersonInImage` present, or
  the `parseType="Resource"` regions)? Worth testing a sidecar with only one of
  the two name mechanisms, or with `stArea:unit` variations.
- **Does the doubling also happen via the desktop app's sidecar path?** Same
  Immich ingestion, so likely yes — but unverified.
- **Can we de-dupe on our side?** After download we could collapse faces with
  identical boxes+person before re-exporting, but that masks rather than fixes.
- **Album ingestion** still untested on the blank server (run `apply_albums`
  / include `immich:albums` and verify).
- **ML path:** if `faceDetection`/`facialRecognition` are *also* enabled, do we
  get the Run-1 pattern (named sidecar faces + unnamed ML faces), or does ML
  now skip faces it thinks are already present? Not yet isolated.

## 8. How to reproduce

```bash
# 1. (optional) pause Immich job queues
IMMICH_SERVER=... IMMICH_API_KEY=... uv run python scripts/pause_jobs.py

# 2. upload the source image (sidecar auto-discovered)
uv run python -m immich_cli upload New/19580128.jpg
#    -> logs asset id, e.g. 6dd908df-...

# 3. release only metadata extraction (keep ML paused) — done in Immich admin UI
#    or via PUT /jobs/metadataExtraction {"command":"resume","force":true}

# 4. download and inspect
uv run python -m immich_cli download 6dd908df-b133-42b8-a0a0-31e81d2ff40d
#    -> downloads/19580128.jpg + .meta.json + .xmp

# 5. check server state directly
uv run python scripts/check_people_cleared.py   # verify/clear people
```

Evidence files in this repo: `trace.log`, `New/19580128.jpg(.xmp)`,
`downloads/19580128.jpg(.meta.json/.xmp)`.
